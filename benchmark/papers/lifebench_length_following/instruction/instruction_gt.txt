Datasets:

    - Specify output length in different units: words, sentences, paragraphs, characters.

- Measure the actual output length (in the specified unit) versus the requested length.
- Compute absolute and relative deviation from the target.
- Analyze compliance rate (fraction of outputs within an acceptable tolerance, e.g., +/- 10%).
- Compare performance across constraint types (upper, lower, exact), length units, and models.

Please design and execute experiments to investigate this research question. Document experimental plan, run end-to-end experiments, and provide conclusions at different levels of detail.

## Experimental Design

1) Main LIFEBench evaluation (Eq/AtMost/AtLeast; 10 lengths; bilingual; 4 tasks)
- Procedure:
  - Load the benchmark instances (10,800 total = 360 base items × 3 control methods × 10 constraints).
  - Control methods: Equal To, At Most, At Least.
  - Length constraints: {16, 32, 64, 128, 256, 512, 1024, 2048, 4096, 8192}.
  - Language-specific unit: English in words; Chinese in characters.
  - Prompting format for every instance:
    - Use the two-part template with both sections present and in this order:
      - [Instruction] … (task-specific request and content)
      - [Requirement] … (control method + length value + unit explicitly named)
  - Inference settings: max_new_tokens = min(65536, provider limit); temperature = 0.8 and top_p = 1 for non-reasoning; reasoning effort = “medium” if supported.
  - Execute in randomized order across all instances to avoid drift.
- What to record:
  - Raw model text output.
  - Language tag, task category/subtype, control method, constraint value.
  - Output word count (unified counter, see Evaluation Details).
  - Computed LD and LS per instance.
  - Empty-output count (exclude those from metrics but log rate).
- Correspondence:
  - Table 1, Figure 3 (Left/Middle/Right), Figures 9–11, E.1–E.2.

3) Discrepancy vs claimed maximum output length

4) Length awareness: self-reported word count
- Procedure:
  - Take the same prompts as in the main evaluation but add this instruction at the end of [Requirement]:
    - “At the end of your response, include the actual total word count of your response, formatted as [WORD COUNT]: a number, where ‘a number’ is the actual number of words you generated.”

  - Extracted “[WORD COUNT]: N” integer (last occurrence if multiple).
  - For each model, fit log–log least-squares regression: log(y) = a·log(x) + b with y = output word count, x = instructed length; fit separately for “real” and “reported” counts.

5) Marker-based length monitoring

- What to record:

## Evaluation Details

- Output length normalization (unified counter):
  - Compute length as: English words + Chinese characters.
  - Chinese characters: count Unicode in [\u4e00-\u9fff].
  - English words: regex \b[a-zA-Z0-9’]+\b (includes contractions and digits).
  - Exclude newlines and tabs from counting.
  - For mixed-language outputs, sum both components.

- Length Deviation (LD):
  - LD = (L_output − L_constraint) / L_constraint.
  - For Equal To summaries, also compute mean absolute LD across constraints.
  - For reporting per E.1:
    - At Most: set LD = 0 whenever L_output ≤ L_constraint.
    - At Least: set LD = 0 whenever L_output ≥ L_constraint.

- Length Score (LS):
  - With k1 = 5, k2 = 2.
  - Equal To (LSE):
    - if LD < 0: LSE = 100 × e^(k1·LD)
    - else:      LSE = 100 × e^(−k2·LD)
  - At Most (LSM):
    - if LD < 0: LSM = 100
    - else:      LSM = 100 × e^(−k2·LD)
  - At Least (LSL):
    - if LD < 0: LSL = 100 × e^(k1·LD)
    - else:      LSL = 100
  - Aggregate LS by arithmetic mean over instances.

- Empty outputs:
  - Exclude from LS/LD aggregates; record empty-output rate.

- Percentiles in max-length study:
  - For each model, compute top-10% and top-25% longest outputs by word count and by token count (see tokenization).
  - Report instructed word target, top-10%/25% achieved words/tokens, and difference vs claim.

- Token counts (for max-length comparisons):
  - Use provider tokenizers for proprietary models; Hugging Face tokenizer for open-source.
  - When only tokens are available, approximate words for instruction setting as words ≈ tokens / 1.33 [uncertain] (specify the exact ratio actually used when running; maintain consistency across models).

- Self-reported count extraction:
  - Regex pattern: \[WORD COUNT\]\s*:\s*(\d+)\b; take the last match if multiple; mark missing if not found.
  - Fit “Real Fit” and “Reported Fit” with numpy.polyfit on log-transformed (x = instructed length, y = count), then back-transform.

- Marker analysis:
  - Find all “[SPECIAL TOKEN]” (exact string match).
  - Compute:
    - Intervals: word counts between consecutive markers.
    - Relative positions: cumulative position / total length.
    - ECDF: proportion of markers per 20%-wide segment.

- Language split:
  - Compute per-language mean LS and mean signed LD per constraint; also compute overall per-language averages.

- Lazy strategies:
  - Keyword lexicons (Table 16) for EN/ZH.
  - Classification:
    - Direct Refusal: contains any keyword and total words < 100.
    - Refusal with Attempt: contains any keyword and total words ≥ 100.
  - Aggregate proportions per constraint and per model.

- Quality judging (K.1/K.2):
  - Judge prompt must return valid JSON with integer scores in {0,…,10}.
  - Record per-dimension integers and their average; compute Pearson r between actual word count and average score (report r and p-value).
  - For scatter plots, use log-scaled x-axis for length.

- Format-constrained counting:
  - Markdown: strip leading header marks (#, ##, …), list markers (-, *, + at line start), inline syntax (*, **, _, __, `…`, [](...)) before counting.
  - HTML: strip all tags via <[^>]+> and decode common entities before counting.
  - LaTeX: strip control sequences beginning with backslash (e.g., \\[a-zA-Z]+(\{[^}]*\})?), environments \begin{…}…\end{…}, and math delimiters ($…$, \[ … \]) before counting.
  - After stripping, apply unified counter.

- Aggregation:
  - Per-instance → per-condition (mean over instances within same model × task × control × constraint) → per-model (mean over tasks/controls/constraints, as specified per figure/table).
  - For Table 1: report mean LS per control; for Equal To also report mean absolute LD.

- Repetition filtering for max-length study:
  - Manual review recommended; optionally flag outputs with high repetition ratio: unique 5-gram ratio < 0.2 [uncertain] or compression ratio (gzip_len / raw_len) < 0.2 [uncertain] for triage; then inspect.


## Implementation Notes

- Prompt templates (must be used verbatim for [Instruction]/[Requirement]):
  - QA:
    - [Instruction] Answer this question: {{content}}
    - [Requirement] Your answer must be {{control_method}} {{length_constraint}} words long.
  - Summarization (choose subtype label accordingly: Encyclopedia/Report/Public Discourse/Academic Article/Book/Dialogue):
    - [Instruction] Summarize this {{subtype}}: {{content}}
    - [Requirement] Your summary must be {{control_method}} {{length_constraint}} words long.
  - Reasoning (Deductive/Inductive/Abductive/Analogical/Cause and Effect/Probabilistic):
    - [Instruction] {{subtype-specific instruction}}: {{content}}
    - [Requirement] Your reasoning must be {{control_method}} {{length_constraint}} words long.
  - Creative Generation (Social Commentary/Prompt-based Creative/Script Expansion/News Writing/Story Expansion/Blog Writing):
    - [Instruction] {{subtype-specific instruction}}: {{content}}
    - [Requirement] The output must be {{control_method}} {{length_constraint}} words long.
  - LABEL:
    - News: [Instruction] You are given a news article excerpt: {{content}}
      [Requirement] Label the article based on its topic. The label must be {{control_method}} {{length_constraint}} words long.
    - Review: [Instruction] Here is a user review: {{content}}
      [Requirement] Provide a label to the review accordingly. The label must be {{control_method}} {{length_constraint}} words long.
    - Tweets: [Instruction] You are given a short text from a tweet: {{content}}
      [Requirement] Provide a label that best represents the tweet. The label must be {{control_method}} {{length_constraint}} words long.
    - Email: [Instruction] The following text is an excerpt from an email: {{content}}
      [Requirement] Provide a label that best categorizes it. The label must be {{control_method}} {{length_constraint}} words long.

- Order of operations:
  - Do not include any counting aids in main evaluation (no [WORD COUNT], no [SPECIAL TOKEN]).
  - Run main benchmark first; then extended constraints; then specialized diagnostics (self-report, markers); finally, format-constrained and judge-based quality assessments.

- Model runtime configuration:
  - Enforce high max_new_tokens; many APIs silently cap outputs—log provider token usage and stop_reason.
  - Reasoning modes: set to medium effort if available; keep temperature at 0.8 only for non-reasoning.

- Output parsing:
  - Strip trailing assistant signatures and safety messages before counting only if clearly outside task (but keep the full raw text archived).
  - For judge JSON, use robust parsing with a fallback sanitizer; if malformed, ask the judge to reformat JSON only (no rescore), preserving the original rationale and numbers.

- Tokenization variability:
  - Token counts vary by tokenizer; record which tokenizer was used per model and keep conversions consistent across the entire run.

- Language unit pitfalls:
  - Chinese must be measured in characters, not words. Mixed-language content is allowed; rely on the unified counter.
  - Some models overuse punctuation or emojis; these are counted as English tokens only if matched by the regex (most will not be), but Chinese punctuation is not counted as characters by the Unicode range.

- Refusals and premature stops:
  - Do not reprompt for compliance in the main benchmark; the goal is to measure first-try adherence.
  - Log refusal classification and keep the original output text.

- Repetition filtering in max-length study:
  - The paper specifies manual filtering. If using heuristics to triage, document thresholds and provide a manual review pass for all flagged cases.

- Input truncation (Summarization sensitivity):
  - Preserve paragraph boundaries when possible; truncate by word count target within the nearest sentence boundary to avoid dangling fragments.

- Regression fits (length awareness):
  - Use log base e; report fitted a, b and R²; if any zero-length outputs occur, exclude from the regression fit and log as missing.

- Resource management:
  - Ultra-long generations are costly. If budget-limited, prioritize: main benchmark → length awareness (self-report) → max-length discrepancy → markers → input-length sensitivity → supplementary sets → format constraints → judge evaluation.

- Optional analyses caveats:
  - EoS logit/prob requires access to logits; skip for APIs that do not expose them.
  - Base-model in-context evaluation requires base checkpoints; otherwise, omit.

- Safety-triggering phrases:
  - The lazy-strategy detector keywords may also be invoked by alignment systems; treat them as behavioral markers rather than hard failures and keep both flagged and unflagged results in aggregates as per Appendix H.

- Randomization and seeds:
  - Shuffle instance order per model; set a fixed random seed for selection/truncation steps to ensure reproducibility.
