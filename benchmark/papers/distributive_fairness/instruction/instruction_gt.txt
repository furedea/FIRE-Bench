## Evaluation Details

- Utilities and feasibility
  - For a parsed outcome (A, p):
    - Utility per person i: ui = vi(Ai) + pi
    - Feasibility: Ai form a partition of the assigned goods (unassigned permitted as “None”); money amounts are integers, nonnegative, and sum to ≤ P (when money is present). Reject outputs that split a good across multiple recipients.

- Property checks (per instance)
  - EF: for all i, j: ui ≥ uj(Aj, pj) evaluated with i’s own utility; here simplified as ui ≥ uj because quasi-linear utilities are person-specific but additively separable and valuations are agent-specific; operationally, check pairwise envy using each agent’s own utility function: vi(Ai)+pi ≥ vi(Aj)+pj for all i, j.
  - EQ* and EQ:
    - Compute ∆(A, p) = max_i ui − min_i ui.
    - EQ* if ∆ = 0.
    - EQ (non-perfect) if ∆ equals the minimum disparity achievable over all feasible outcomes for the instance. Compute the minimum disparity by exhaustive enumeration (see “Enumeration” below).
  - RMM:
    - Let m* = max over all feasible outcomes of the minimum utility min_i ui. Outcome is RMM if min_i ui = m*.
  - USW:
    - Let s* = max over all feasible outcomes of sum_i ui. Outcome is USW if sum_i ui = s*.
  - PO:
    - Outcome is Pareto optimal if there does not exist any other feasible outcome (A′, p′) such that ui′ ≥ ui for all i and ui′ > ui for at least one i. Decide by exhaustive dominance test against the enumerated outcome set.
  - Other: none of the above satisfied.

- Enumeration for ground-truth property thresholds
  - Goods-only:
    - For m goods and n persons, enumerate (n+1)^m assignments (the extra “+1” is for discarding).
  - Goods+money:
    - Enumerate all goods assignments as above, and all integer money splits across n persons with total money k ∈ [0, P]. The number of money splits equals sum_{k=0..P} C(k+n−1, n−1) = C(P+n, n).
  - For each feasible outcome, compute utilities, disparity, min-utility, sum-utility once; cache global maxima/minima for EQ, RMM, USW, and PO checks.

- Output normalization and parsing
  - Normalize person labels to integers based on “Person i” strings in JSON; normalize “None” (case-insensitive) as discarded.
  - Money keys must be “Person i money”; coerce to int; if missing in goods+money instances, treat as 0. If total exceeds P, mark as invalid.
  - If Stage-2 JSON is malformed:
    - Attempt regex fallback to extract per-good assignees and per-person money.
    - If still unresolved, mark sample as Invalid and exclude from property checks; track Invalid count.

- Aggregation
  - Per-sample: record property flags; if multiple properties hold (e.g., EF+PO), set all relevant flags.
  - Per-condition (instance × prompt variant): compute proportions for each property or property-combination of interest, plus “Other” and “Invalid”.
  - Overall across instances: for each model/condition, compute the mean percentage per notion across instances and 95% confidence intervals using instance-level means:
    - Mean = (1/K) Σ_k p_k
    - CI (normal approx) = mean ± 1.96 × (sd(p_k)/√K), where K is number of instances in the aggregate (e.g., K=10 for I1–I10).
  - For selection-from-menu tasks, aggregate per-option selection rates and map each option to its known notions.

- Statistical tests and effect sizes
  - Within-instance comparisons of distributions between two conditions (e.g., baseline vs template):
    - For property vs not (2×2), use Fisher’s exact test (scipy.stats.fisher_exact, two-sided).
    - For multi-category distributions (e.g., five-option menu), use chi-square test of independence (scipy.stats.chi2_contingency). Report Cramér’s V = sqrt(χ²/(N·(min(r−1,c−1)))).
  - Report significance at p < 0.05. Mark changes as “significant increase/decrease” consistent with Tables 10–13 and Figure 11 logic.

- Success rates for targeted-notion tasks
  - Persona/objective/refinement: success = allocation satisfies the intended notion (EQ*, EF, RMM, or USW).
  - Refinement: report success within ≤2 retries, and per-attempt success curve.

- Inequality-disparity choices in unfair menus
  - Summarize the frequency of choosing the minimum-disparity option vs higher disparities; compute rank-biserial correlation between disparity level and choice probability [optional].

- Handling “bystander vs participant” role
  - For each role condition, compute distribution over properties and the decision-maker’s payoff. For each instance, compute the difference in the decision-maker’s expected utility vs bystander baseline and test with a two-sample t-test over samples [uncertain; non-normal; alternatively use Mann–Whitney U].


## Implementation Notes

- Core prompts and templates
  - Use the exact wording for the final instruction lines:
    - Baseline: “Your task is to determine the allocation that you consider to be the fairest - there are no ‘correct’ or ‘incorrect’ answers.”
    - Modified intentions (F.1): replace last line accordingly (“most desirable”, “acceptable by all”, “an allocation”).
    - Personas/objectives: replace last line per J.3 (“you only care about minimizing the inequality...” or explicit objective statements).
  - Two-stage prompting must be used for the baseline unless explicitly testing the template-based variant. This separation is required to avoid biasing model choices with output templates.

- Output-parsing logic
  - Require Stage-2 JSON to include all goods; missing goods default to “None”.
  - Enforce integer money; if decimals appear, round down to nearest integer; flag as “Adjusted” and record separately.
  - If a good is assigned to multiple persons or split, mark Invalid.
  - Map “You” (I9/I10 prompts) to Person 1 for scoring consistency.

- Money handling
  - The paper evaluates money as a transferable divisible resource with integer units. Enforce integer splits and allow unallocated balance (Σ pi ≤ P).
  - For benchmarks I1.1–I1.4, check equivalence against Table 45 characterizations:
    - EQ*+EF+USW: specific (p1, p2) per instance
    - EQ*+EF: ranges per instance
    - EF: ranges per instance
    - USW: any (p1, p2) with p1+p2 ≤ 50 maximizing total utility

- Enumeration performance
  - Goods-only with up to 6 goods and 3 persons: (3+1)^6 = 4096 outcomes; trivial.
  - Goods+money with up to 3 goods and P ≤ 9: ≤ 64 goods assignments × C(P+3,3) money splits (≤ 64×220=14,080); acceptable.
  - For I1.1–I1.4 (2 goods, P=50, 2 persons): 3^2 × C(52,2) = 9 × 1326 ≈ 11,934 outcomes; acceptable.

- Order of operations
  - Baseline (two-stage) runs first on I1–I10 to establish distributions.
  - Then run:
    - Money benchmarks (I1.1–I1.4)
    - Magnified-inequality (I2*, I4*)
    - Selection menus (human-derived; unfair options)
    - Context-augmented menus
    - CoT prompting
    - Modified intentions
    - Personas/objectives
    - Refinement
    - Cognitive-bias role swaps
    - Robustness to ordering and to template-based prompting
  - Do not mix CoT, templates, or personas into the baseline unless the variant is being explicitly tested.

- Pitfalls/caveats noted in the paper
  - Models sometimes deploy greedy heuristics (e.g., assign each good to the highest valuer) or round-robin-like allocations; these can be EF/USW but typically not EQ*. Ensure property checks are computed against enumerated ground-truth maxima/minima and not inferred from explanations (Appendix K).
  - Template-based prompting can bias models toward goods-centric decisions (USW or discard-low-value goods). Keep template variants isolated.
  - Ordering sensitivity (goods/person list order) can change distributions; ensure the three derived variants (I′, I′′, I′′′) are generated by deterministic permutations and labeled consistently for analysis.
  - When money is present, some models may omit money splits; treat missing as zeros and score; also record the fraction of samples that discard all money.

- Reproducibility
  - Fix the set of prompts exactly as in J.1–J.6 (copy text verbatim).
  - Keep temperature fixed at 1.0 across all conditions.
  - For selection menus, present options in the exact order listed (do not shuffle) to match the paper; when testing order sensitivity, do so as a separate manipulation.
